1. Show HN

Title (max 80 chars)

Show HN: NAC3 -- web UIs that humans and AI agents see the same way

Alternative if "AI" feels overused on the day:

Show HN: NAC3 -- a contract between web UIs and the things that drive them

Body

For 40 years we have built UIs assuming one species of operator:
humans, with a mouse and a keyboard, looking at a screen. The
metaphor has not changed since 1984.

We are about to add a second species -- AI agents, voice runners,
RPA bots, screen readers, every assistive tool you can name. Each
one is currently guessing what the UI means from screenshots, raw
DOM, accessibility trees, brittle selectors. They guess wrong,
often, expensively.

NAC3 (Native Agent Contract) is a small public spec that lets a
web UI declare itself: "this is the save button. This is the verb
it accepts. This is the event it emits when finished." A human
clicks it. An LLM calls it by name. A screen reader announces it.
A Playwright spec dispatches it. All five see the same surface.

It is what Captain Picard's interface has that ours does not: a
contract between the human and the machine, on shared ground. The
captain says "warp speed" and the ship executes -- not because of
voice magic, but because the ship's surface is contractually
addressable. NAC3 is that surface for web UIs.

Why this is more than buzzwords:

  <button data-nac-id="invoice.save"
          data-nac-role="action"
          data-nac-action="save">Save</button>

  // Any caller, any locale, any device:
  await NAC.click('invoice.save')
  // -> emits 'nac:action:succeeded' when the side effect completes.

That is the entire surface. About 80 lines of HTML attributes for
a typical mid-size app. ARIA still works. Frameworks still work.
NAC3 is the missing layer underneath everything that wants to
drive the page.

Side effect we did not initially design for: the same contract
also solves screen reader compatibility, switch access, voice
control, and eye tracking with zero extra code. The first web
spec built deliberately so that humans, AI, and assistive
technology share the same operating surface.

Spec + reference runtime are public today, Apache 2.0 (spec) +
MIT (runtime). Eight live demos in any browser. The LLM
intermediary in the demos is wired to real Claude Sonnet -- not a
local pattern matcher pretending to be smart. Try it before
believing anything we say.

  npm install @nac3/runtime

  Demos:    https://yujin.app/nac-spec/
  Spec:     https://yujin.app/nac-spec/SPEC.md
  GitHub:   https://github.com/pkuschnirof/rpaforce-crm

Governance: currently stewarded by Yujin, committed in writing to
moving to a neutral foundation (W3C community group, Linux
Foundation, or equivalent) when adoption justifies neutral
governance. Apache + MIT guarantee the spec survives whatever
happens to us, separate from any promise we make.

This launch is half announcement, half invitation. Star Trek
interfaces require a contract. Here is one. Adopt it, fork it,
improve it. Open issues, open PRs, open opinions.

Happy to answer anything.

Pre-emptive reply drafts (top expected HN objections)

Objection: "This is just ARIA reinvented."

> NAC3 layers verbs + ack events + provenance on top of ARIA. ARIA > tells the screen reader the element is a button; NAC3 tells the > caller what verb the button accepts, what event it emits when the > side effect completes, and whether the click came from a real > user (via isTrusted). ARIA + NAC3 coexist on the same element. > The two communities reuse each other's work, not compete. Many > adopters end up with both role="button" and > data-nac-role="action" on the same DOM node.

Objection: "Why would I add 80 lines of attributes when Claude will eventually figure out my page from a screenshot?"

> Two reasons. (a) Claude figuring out your page from screenshots > costs serious tokens per describe; declaring the surface > contractually costs zero tokens at runtime. The economics flip > past one user. (b) "Eventually" means "after weeks of fixing > wrong-element clicks in production." An explicit contract > eliminates that class of bug categorically. Determinism beats > intelligence when determinism is cheap.

Objection: "Yujin owns the standard, why should I bet on it?"

> Apache 2.0 spec + MIT runtime. You can fork either tomorrow. We > stated in writing (SPEC.md section 13.5) that we will move > governance to a neutral foundation when adoption justifies it. > The license itself guarantees you survive us, separate from any > promise we make. The standard outlives any single steward.

Objection: "How does this work with native iOS/Android?"

> v2.2 covers web + WebView (which includes a large share of > mobile, since most apps that look native are actually WebView > wrappers). Native UIKit / Jetpack Compose is on the v3.0 > roadmap, mapping to accessibilityIdentifier (iOS) + > contentDescription (Android). The OS accessibility APIs already > give us 70% of what NAC3 needs natively.

Objection: "The Captain Picard analogy is cute, but where's the moat against Anthropic shipping a better DOM-reading agent?"

> NAC3 is not betting that DOM agents fail. We are betting they > succeed AND that declaring the surface costs less than guessing > it. ARIA did not lose to better screen readers; ARIA + better > screen readers together work better than either alone. NAC3 > plays the same role for agents. If Anthropic ships a 10x better > agent tomorrow, that agent works 10x better on NAC3-compliant > sites, not less.

Objection: "What about the accessibility angle? Is that the real pitch?"

> No. The accessibility win is a side effect we noticed midway > through, not the pitch. NAC3 is a contract layer for any caller. > Screen readers happen to be a caller. Voice control happens to > be a caller. Switch access happens to be a caller. They all win > because the contract is caller-agnostic. We mention > accessibility because it is real and because it positions us > well for EU EAA and US ADA Title III compliance scenarios, but > the standard would exist even if assistive tech did not.

Today we are publishing NAC3 -- the Native Agent Contract -- as a

The premise: when Captain Picard says "warp speed," the ship

surface is contractually addressable. NAC3 is that surface for

screen reader announces it. A Playwright spec dispatches it. All
five see the same operating surface.

Why this is more than a slogan: the technical change is small.
About 80 lines of HTML attributes for a typical mid-size app.
ARIA still works. React, Angular, Vue, vanilla still work. NAC3
sits underneath, addressed by name rather than by selector or
visual coordinate.

Side effect we did not initially design for: the same contract
solves screen reader compatibility, switch access, voice control,
and eye tracking with zero extra code. The first web spec built
deliberately so that humans, AI, and assistive technology share
the same operating surface. For enterprises this materially
changes the compliance posture for the EU Accessibility Act (June
2025) and the rising tide of US ADA Title III web lawsuits.

For RPA teams: automation no longer breaks every UI redesign.
Selectors die; ids survive. Three integration guides for UiPath,
Automation Anywhere, Blue Prism, and Playwright are published
today.

For accessibility advocates: this is the largest single expansion
of accessibility-by-default we have seen since ARIA shipped. We
owe the ARIA community everything. NAC3 layers on top, it does
not replace.

# Section 1 -- The mouse paradigm is 40 years old (~300 words)

The mouse + keyboard interface paradigm has not meaningfully
changed since 1984. We have built more beautiful versions of the
same metaphor. More fonts, more shadows, more animations. Same
metaphor.

Meanwhile we are adding a second species of operators to every
web app on Earth: AI agents, voice runners, RPA bots, screen
readers, switch access. Each is currently guessing what the UI
means. Each guesses differently. Each is wrong sometimes. The
collective cost across the industry is enormous.

We need a contract.

# Section 2 -- The Captain Picard analogy (~200 words)

When Captain Picard says "warp speed", the ship executes. Not
because of voice magic. Because the ship's operating surface is
contractually addressable. There is a known set of verbs the ship
accepts. There is a known set of events the ship emits when each
verb completes. The captain does not need to navigate to a panel
or grep for a UI element. He addresses the ship by name.

That is the kind of interface we owe to every operator of every
web UI -- whether the operator is a human, an LLM, or a switch
device controlled by an eye blink. NAC3 is that contract.
