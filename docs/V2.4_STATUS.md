and did not race two agents on the same UI.

### V24-05 — agent authority + auto-interposition

**NOT implemented in the runtime.** Only RFC draft exists (see
`docs/RFC_V24-04.md`; V24-05 will live in `rfcs/RFC-V24-05-agent-authority.md`
when drafted).

Required for 2.4 (per `brief-testear-2.4.md` Phase B):

- `NAC.confirm_action({ action, payload, justification })` API.
- `NAC.request_authority({ scope })` / `release_authority()`.
- `NAC.set_confirm_handler(fn)` for host interposition.
- "Sudo model" semantics: agent emits an action, runtime interposes,
  host confirms (UI prompt or auto-approve handler), then runtime
  dispatches.

Estimated implementation: 1-2 focused days.

## What testing v2.4 still requires

Per `brief-testear-2.4.md`:

### Fase A.3 — Hash determinism test (BLOCKING gate)

Before any v2.4 work proceeds:

```bash
node test/hash_determinism.test.js
# 100/100 hashes identical for identical input
```

If non-deterministic, the canonical serializer has issues (key
ordering, Unicode escape, etc) — fix before continuing.

This test does NOT exist yet. Writing it is the first thing v2.4
testing needs.

### Fase B — V24-05 implementation

Per the diagnóstico-RFC-V24-05.md (7 fixes applied to v2 draft).
Not started.

### Fase C — Concurrency + authority benchmark

3 concurrency tasks + 3 authority tasks the current benchmark does not
exercise:

**Concurrency (V24-04 enforcement):**
1. Two agents racing on the same invoice; the second one to dispatch
   should see `snapshot_stale` and retry.
2. Agent dispatch interleaved with host re-render; `expected_plugin_version`
   should fire correctly across the re-render boundary.
3. Hydration race: agent dispatches before `markHydrationComplete()`;
   runtime queues or rejects per spec.

**Authority (V24-05):**
4. Destructive verb (`cancel_invoice`) gated by `confirm_action`;
   measure host confirmation latency, agent retry pattern after rejection.
5. Persistent authority scope (`request_authority` for a session);
   verify dispatches within scope skip confirmation.
6. Authority delegation: parent agent grants scope to child; child
   dispatches under inherited authority.

These tasks ARE NOT in the current bench harness. The harness needs:

- Multi-agent dispatch loop (today it's single-agent).
- Concurrency-aware fixture re-renders (today static after setup).
- Host confirmation handler simulation (yes/no, latency, timeout).
- Authority-scope tracking across multi-dispatch.

### Fase D — Publish 2.4.0-rc1 with `npm publish --tag next`

NOT `latest`. Adopters install with `npm install @nac3/runtime@next`.

## Recommended sequencing (per Claude Opus's vote)

1. **Now → +2 weeks:** let 2.3.0 stable settle. Watch for adopter
   feedback on `syncPlugin` + decorations + `plugin_version` field shape.
2. **+2 weeks:** review what early adopters discovered. Decide whether
   to bump enforcement to opt-out (vs opt-in) for v2.4, or whether to
   refine the semantics first.
3. **+2-4 weeks:** start Fase A.3 (hash determinism test), then B.
4. **+4-6 weeks:** Fase C (concurrency benchmark).
5. **+6 weeks:** publish 2.4.0-rc1 as `@next`. Soft launch.
6. **+8 weeks:** if no critical issues, 2.4.0 stable as `@latest`.

This is a soft target, not a deadline. If V24-05 reveals design
issues during implementation, slip the schedule rather than ship
half-baked authority.
