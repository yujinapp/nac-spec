# تأثير NAC3 على الاختبار وضمان الجودة

**إصدار NAC3:** 2.2 مستقر.
**الجمهور المستهدف:** مهندسو الاختبار، قادة ضمان الجودة، مهندسو أتمتة الاختبار (SDETs)، والمديرون التنفيذيون للتقنية (CTOs) الذين يقيّمون التكلفة طويلة الأمد لصيانة الاختبارات عند اعتماد NAC3.

## ملخص سريع

كود الاختبار الذي يستخدم معرّفات NAC3 يصمد أمام إعادة تصميم واجهة المستخدم. أما كود الاختبار الذي يعتمد على محددات CSS فلا يصمد. هذه الخاصية وحدها تغيّر اقتصاديات صيانة الاختبارات من "تكلفة تتناسب طردياً مع تغييرات الواجهة" إلى "تكلفة تتناسب طردياً مع تغييرات الميزات" -- وهو ما يعني عادةً تقليل الجهد بمقدار 5 إلى 10 أضعاف.

## حسابات الصيانة في الوضع الراهن

تحتوي مجموعة اختبارات Selenium / Cypress / Playwright النموذجية لتطبيق ويب غير بسيط على مئات المحددات:

```ts
await page.locator('button.btn-primary.btn-large:has-text("Save")').click();
await page.locator('div.modal > div.body > input[type="text"]:first-of-type').fill('Acme');
await page.locator('table.invoice-grid > tbody > tr:nth-child(1) > td:last-child > button').click();
```

تنكسر هذه المحددات عند:

- إعادة تسمية فريق التصميم لـ `.btn-primary` إلى `.btn-cta`.
- إضافة عنصر div مُغلِّف لأغراض إمكانية الوصول.
- تدويل نص الزر بحيث يصبح "Save" هو "Guardar" في اختبارات المستأجر الإسباني.
- تحويل تخطيط الشبكة إلى `grid-template-rows`.
- أي تغيير في الصفحة لا يمسّ المقصد الدلالي.

تُقدّر الدراسات الاستقصائية في الصناعة (2024-2025) أن **30-50% من وقت مهندس ضمان الجودة يُصرف في صيانة المحددات**. ويزداد هذا الرقم سوءاً كلما نما التطبيق.

## حسابات الصيانة مع NAC3

```ts
await NAC.click('invoice.save');
await NAC.fill('invoice.client_name', 'Acme');
await NAC.click('invoice.line.42.delete');
```

هذه الاستدعاءات تصمد في مواجهة:

- إعادة تسمية أصناف CSS (لأن المحددات لا تشير إلى CSS).
- إعادة هيكلة شجرة DOM (لأن المحددات لا تشير إلى البنية).
- تغييرات تسميات التدويل (لأن المحددات لا تشير إلى النصوص).
- الانتقال من تخطيط Grid إلى Flex.
- استبدال مكتبات المكوّنات.

ولا تنكسر إلا عند:

- إعادة تسمية فريق المنتج لفعل ما (`save` -> `commit`).
- حذف زر بالكامل.

هذه **تغييرات على مستوى الميزات**، لا على مستوى الواجهة. يحتاج الاختبار إلى التحديث للسبب ذاته الذي يحتاج فيه كود الإنتاج إلى التحديث. وهذا هو الأساس الصحيح للتكلفة.

من البيانات الداخلية لـ Yujin CRM (2025):

| المقياس | قبل NAC | بعد NAC | الفرق |
|--------|-----------|-----------|-------|
| متوسط أسطر مواصفات Playwright | 187 | 64 | ‎-66%‎ |

الرقم 89% هو الأبرز. **الغالبية العظمى من اختبارات NAC3 تستمر في العمل خلال التطور الطبيعي للمنتج**، في حين تتدهور نظيراتها القائمة على المحددات تدريجياً.

## ما يتيحه NAC3 لأتمتة الاختبار

### 1. مجموعة اختبارات مستقرة

اختبار كُتب عام 2024 يستدعي `NAC.click('invoice.save')` سيظل يعمل في 2026 طالما بقي الفعل `save` ضمن خارطة طريق المنتج، حتى لو أُعيد بناء DOM المحيط بالزر ثلاث مرات.

### 2. اختبار متعدد المتصفحات دون تبديل أوضاع المحددات

تتصرف محددات CSS بشكل مختلف عبر Chromium / Firefox / WebKit في الحالات الحدية (العناصر الزائفة، حلقات التركيز، shadow DOM). يُوزَّع NAC3 عبر محلّل وقت التشغيل -- مسار الكود ذاته بصرف النظر عن المتصفح.

### 3. اختبارات مستقلة عن التدويل

في التطبيقات متعددة اللغات: تحتاج مجموعة الاختبارات اليوم إلى تشغيلات منفصلة لكل لغة لأن "Save" / "Guardar" / "Speichern" كلها تمثّل الزر ذاته. مع NAC3 يستدعي الاختبار المعرّف، ويحلّ وقت التشغيل عبر اللغات. **تكتب اختباراً واحداً، ويعمل على 10 لغات** (واحدة لكل ).

### 4. تأليف الاختبارات بمساعدة LLM

نموذج LLM يرى `NAC.describe()` يمكنه إنتاج مواصفة اختبار كاملة من وصف نثري: "اختبر أن إضافة صف ثم حذفه يُعيد الجدول إلى حالته الأولية." يُصدر النموذج استدعاءات `NAC.*`؛ تراجعها وتُدمجها. لدى Yujin CRM نحو 250 مواصفة أُلِّفت بهذه الطريقة وخضعت للمراجعة قبل الدمج.

### 5. الاختبارات ذاتية الإصلاح عبر الاكتشاف

عندما يفشل اختبار بسبب إعادة تسمية معرّف:

```ts
try {
  await NAC.click('invoice.save');
} catch (e) {
  if (e.code === 'not_found') {
    // Re-discover; the verb 'save' may live under a new id.
    const r = await NAC.click_by_verb('invoice', 'save');
    if (r.ok) console.warn('id has changed; update test');
  }
}
```

تمنحك `click_by_verb` في وقت التشغيل احتياطاً ذاتي الإصلاح يُظهر "هذا الاختبار يحتاج تحديثاً، لكن الإجراء لا يزال يعمل" -- وهو وضع فشل أفضل بكثير من "المحدد غير موجود، توقف تام".

### 6. توليد الاختبارات من المانيفستات

`NAC.validate_global({probe: true})` يُنشئ نقرة على كل عنصر `role="action"` ويتحقق من إصداره لحدث الإقرار القياسي خلال 5 ثوانٍ. **هذا اختبار دخان مُولَّد تلقائياً لكامل السطح القابل للنقر في التطبيق**. شغّله في CI؛ يكتشف أي زر يُركَّب دون إصدار إقرار صحيح.

### 7. تغطية خط الأنابيب حسب المرحلة

تُنظّم مجموعة الاختبارات المرجعية لـ Yujin (NAC_TEST_MANUAL.md) الاختبارات حسب مراحل خط أنابيب NAC3:

- المرحلة 1 (إدخال STT)
- المرحلة 2 (إزالة الغموض)
- المرحلة 3 (وسيط LLM)
- المرحلة 4 (استدعاءات `NAC.*`)
- المرحلة 5 (التأثير الجانبي على DOM)
- المرحلة 6 (حدث الإقرار)

تُقاس التغطية **لكل مرحلة**، لا لكل سطر كود فحسب. يُفيد التقرير المرجعي لـ Yujin بمتوسط مرجّح يبلغ نحو 95% عبر جميع المراحل. اعتماد هذا المخطط يمنحك بطاقة تغطية تتوافق مباشرة مع العقد.

## الأثر على أطر الاختبار الموجودة

### Playwright

تكامل مباشر. `page.evaluate()` يستدعي `NAC.*`. تبقى المحددات كاحتياطي لتأكيدات التخطيط. يتضمن المرجع Yujin 16 مواصفة Playwright في `tests/e2e-nac/specs/`.

### Cypress

`cy.window().then(win => win.NAC.click(id))`. النمط ذاته. تُغلّف الأوامر المخصصة استدعاءات NAC:
`cy.nacClick('invoice.save')`.

### Selenium

منفّذ JavaScript: `driver.execute_script('return window.NAC.click(arguments[0])', 'invoice.save')`.

### Jest + React Testing Library

```ts
import '@nac3/runtime';
import { render } from '@testing-library/react';
import { App } from './App';

test('save works', async () => {
  render(<App />);
  await window.NAC.click('invoice.save');
  expect(mockApi.save).toHaveBeenCalled();
});
```

يعمل NAC3 جنباً إلى جنب مع React Testing Library، لا في مواجهته.

### Karma / Jasmine / أطر التشغيل القديمة

حقن مباشر عبر `window.NAC`. أي إطار قادر على تشغيل JavaScript في سياق متصفح يعمل.

## تكلفة الاعتماد

### تطبيق قائم

وفقاً لـ [دليل الترحيل](AI_PLAYBOOK_MIGRATION.md)، توقّع:

- نحو يوم واحد لكل شاشة للتزيين وإنشاء المانيفست.
- نحو يوم واحد لكل شاشة لترحيل مجموعة الاختبارات.
- الإجمالي لتطبيق مكوّن من 20 شاشة: نحو 6 أسابيع من وقت مهندس واحد، تُسترد من خلال وفورات الصيانة خلال 3-4 أشهر.

### تطبيق جديد

مدمج من البداية. يعامل دليل التطوير من الصفر سمات NAC3 باعتبارها اهتماماً من الدرجة الأولى. لا تكلفة تحديث.

## المخاطر والتخفيف

### الخطر -- "لا نثق في الاختبارات المُولَّدة بواسطة LLM"

مشروع. يُنتج النموذج مرشحاً؛ يراجعه إنسان ويدمجه. نفس سير العمل مع Copilot. المجموعة التي تُشحن هي بالضبط ما وافق عليه الفريق، لا ما كتبه النموذج.

### الخطر -- "ستتحول معرّفات NAC إلى دَين تقني مع الوقت"

صحيح إن تركتها تتدهور. عامل معرّفات NAC كأسماء أعمدة قاعدة البيانات: أعد التسمية عبر ترحيل، ولا تحذف أثناء التشغيل. يكتشف CLI الخاص بـ `@nac3/runtime` المعرّفات اليتيمة عبر الفحص الساكن.

### الخطر -- "ماذا لو توقف اعتماد NAC؟"

المواصفة مرخّصة بـ Apache-2.0. وقت التشغيل أقل من 200KB. في أسوأ الأحوال: تمتلك الأداة، وتبقى المعرّفات مستقرة. حتى أسوأ الأحوال أفضل من محددات CSS.
