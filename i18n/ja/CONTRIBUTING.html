NAC3 へのコントリビューション

仕様バージョン: 2.2 stable（+ v2.3 相互運用プレビュー）

ガバナンス

NAC3 は現在 Yujin によって管理されています。仕様は Apache 2.0、リファレンスランタイムは MIT ライセンスです。Yujin は、採用状況が中立的なガバナンスを正当化する場合、NAC3 を中立的な財団（W3C コミュニティグループ、Linux Foundation、または同等の組織）に移管することを約束しています。それまでの間、仕様の変更は以下の RFC プロセスに従い、公開 API またはワイヤーフォーマットへの変更には少なくとも 14 日間のパブリックコメント期間が設けられます。

Apache 2.0 + MIT のライセンス構成により、Yujin の企業状況に変化が生じた場合でも、仕様とランタイムは存続します。両ライセンスのもとでのフォークは明示的に歓迎されています。

コントリビューションをご検討いただきありがとうございます。NAC3 は公開仕様とリファレンス実装で構成されており、どちらもコントリビューションを受け付けています。

コントリビューションの種類

1. 仕様変更（RFC 必須）

SPEC.md、ABOUT.md、または docs/NAC_V*_ROADMAP.md の編集は仕様変更に該当します。PR を開く前に以下の手順を踏んでください：

  1. RFC: <一行の概要> というタイトルで GitHub Issue を作成する。
  2. 問題の種類を説明する（修正するバグや制限、できれば具体的な再現手順を含む）。
  3. 提案するコントラクト変更を説明する。
  4. 既存の採用者向けの移行パスを説明する。
  5. PR を開く前に、少なくとも 1 人のメンテナーから Issue への返信を待つ。

RFC Issue と対になっていない仕様 PR は、このセクションへのポインターとともにクローズされます。

2. リファレンスランタイムの変更

js/nac.js、js/nac-v2-extensions.js、または js/nac-chat-client.js の編集。以下の場合は RFC なしで PR を歓迎します：

  • 変更がランタイムを現在の仕様に合わせるバグ修正である場合。
  • 変更が動作に影響を与えないパフォーマンス改善である場合。
  • 変更がドキュメント、型定義、またはテストカバレッジに関するものである場合。

仕様コントラクトに影響するランタイムの動作変更を伴う PR は、必ず先に仕様 RFC と対にする必要があります。

3. デモ、ツール、またはドキュメントの改善

example*.php、tools/、guides/、またはその他の非仕様 Markdown の編集。直接 PR してください。変更は最小限に留めてください。大きな PR 1 つより小さな PR 10 個を推奨します。

コードスタイル

  • ソースファイルは ASCII のみ（プロジェクトは GoDaddy にデプロイされており、PHP 8.3 はコメント内でも非 ASCII を拒否します）。em ダッシュには -- を使用し、-- は使わないでください。
  • JS: ランタイムファイルにトランスパイラー、バンドラー、ビルドステップは使用しない。プレーンな ES2018+。npm パッケージは同じソースに ESM/CJS ラッパーを追加します。
  • PHP: heredoc はシンプルに保つ（{$var} のみ、式は不可）。
  • コメント: WHY（なぜ）を説明し、WHAT（何を）は書かない。差分が「何を」は示しています。
  • テスト: 動作変更にはすべて、変更前に失敗し変更後に通過するテストを付ける。プッシュ前にリポジトリルートから make test-launch を実行してください。

コミットスタイル

「fix(nac): treat tab role drift as register-time error」のように書き、「Fixed tab thing」のような書き方はしない。

  • サブジェクトは 70 文字以内、現在形の命令形で。
  • 本文には問題、原因、修正内容を説明する。関連コミットは短い SHA で参照する。
  • AI を活用したコミットへの Co-author トレーラーは問題ありません。ツールの使用を隠す必要はありません。

レビュー

  • バグ修正 PR: 承認者 1 名でマージ。
  • ランタイム／仕様 PR: 承認者 1 名 + CI グリーンでマージ。
  • 仕様変更 PR: 対になる RFC Issue でのディスカッション + 承認者 1 名 + CI グリーン + PR オープン後 7 日間のコメント期間。

ライセンス

PR を提出することで、プロジェクトに合わせて Apache-2.0 のもとでコントリビューションをライセンスすることに同意したものとみなされます。PR テンプレートにチェックボックスが含まれていますので、チェックを入れてください。

行動規範

技術的に正確に、簡潔に、そして親切に。意見の相違は問題ありませんが、人身攻撃は許容されません。繰り返し違反した場合、メンテナーはスレッドをクローズしたり、コミットアクセスを取り消したりすることがあります。

質問する場所

  • 設計に関する質問、「NAC3 をこの用途に使うべきか？」、ショーケースには GitHub Discussions を使用してください。
  • バグ報告には GitHub Issues を使用してください。
  • セキュリティの開示には nac@yujin.dev へ連絡してください（SECURITY.md 参照）。
