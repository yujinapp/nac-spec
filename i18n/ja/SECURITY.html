NAC3 -- セキュリティモデル

仕様バージョン: 2.2 stable（+ v2.3 相互運用プレビュー）

脅威モデル

NAC3 はエージェントと UI の間に位置します。これは認証レイヤーではなく、コントラクトレイヤーです。複数の異なるトラストバウンダリが存在します。このドキュメントでは、NAC3 が保護するバウンダリとそうでないバウンダリを明確に区別するために、それぞれを定義します。

バウンダリ A: ユーザー -> UI

NAC のスコープ外です。既存の認証機構（セッション、OAuth、SSO、MFA）を使用してください。ユーザーが認証済みであれば、NAC3 は UI 上でそのユーザーが実行できるあらゆる操作を許可されたものとみなします。

バウンダリ B: ユーザー制御エージェント -> UI

ユーザーがエージェントに対して、自分のブラウザセッションを操作する権限を付与するケースです。例として、音声アシスタント、スクリーンリーダー、同一ページに埋め込まれた LLM チャットクライアントなどが挙げられます。このバウンダリにおける NAC3 の役割は以下のとおりです。

  1. エージェントに安定した要素名を提供し、座標のスクレイピングなしに操作できるようにする。
  2. event.isTrusted を公開し、ホストがセキュリティ上重要な操作（支払い、削除、ロール付与）に対して合成クリックを拒否できるようにする。エージェントは isTrusted=true を偽造できません。実際のユーザージェスチャーのみがこれを設定します。
  3. イベントレベルの確認応答を提供し、エージェントが DOM を再読み込みせずに完了した操作を把握できるようにする。

NAC3 は、ユーザーが明示的に信頼したエージェントがその信頼を悪用するケースを防ぎません。それはユーザーエクスペリエンスの問題（重要な操作前の同意プロンプト）であり、NAC ではなくアプリケーション側で対処するものです。

バウンダリ C: 外部サービス -> UI（LLM 仲介）

ユーザーの音声プロンプトがリモート LLM に送信され、LLM が NAC3 アクションを返す場合、LLM はトラストプリンシパルになります。このバウンダリにおける NAC3 の役割は以下のとおりです。

  1. LLM は NAC.describe() が公開する情報（ツリースナップショット + 登録済みマニフェスト）のみを参照します。ユーザーの認証トークン、Cookie、またはマニフェストで宣言された範囲を超えた DOM の内部情報は参照できません。
  2. LLM はクリックを直接引き起こすことはできません。LLM は構造化されたアクションを返し、チャットクライアントがディスパッチ前にそれを検証します（nac_id は存在するか？動詞は許可されているか？）。
  3. チャットクライアントは、送信したスナップショットに含まれていない nac_id を持つアクションを拒否すべきです（SHOULD）。これにより、任意の ID を埋め込むプロンプトインジェクションを防ぎます。

NAC3 は LLM のプロンプトテンプレート、レート制限、またはフィルタリングを規定しません。推奨事項については guides/LLM_WIRING.md を参照してください。

バウンダリ D: テナント -> テナント（マルチテナント展開）

テナントがランタイムを共有しつつデータを共有しないマルチテナント SaaS の場合、NAC3 は HMAC 署名付きマニフェストでこれを保護します。

  1. 各テナントは、サーバー側に保存されたテナントごとのシークレットを使用して、安定したシリアライゼーションに対して計算された HMAC 署名とともにマニフェストを送信します。
  2. ランタイムは NAC.register() 時に、アクティブなテナントに期待するシークレットを使って HMAC を再計算します。署名が一致しない場合、マニフェストは拒否されます。
  3. 悪意のあるテナントは、署名シークレットなしに別のテナントのマニフェストを偽造できません。

NAC3 は、基本的なサイズ上限を超えた過度に大きいまたは不正なマニフェストの登録を防ぎません。信頼できないマニフェストを受け入れる場合は、サーバー側でマニフェスト登録のレート制限を設けてください。

バウンダリ E: 悪意のあるスクリプト -> ページ

攻撃者が制御する JS（XSS、サプライチェーン侵害）を含むページはすでに危険な状態です。NAC3 はここでは対処できません。攻撃者は NAC.click(...) を直接呼び出すことができます。CSP、SRI、および通常の Web セキュリティスタックで緩和してください。

プロベナンスシグナル

成功イベントにおける is_trusted

すべてのアクション成功イベントの detail には is_trusted: boolean が含まれます。ホストはセキュリティ上重要な操作に対してこれを必須にできます。

document.addEventListener('nac:action:succeeded', function (e) {
  var d = e.detail;
  if (d.action_id === 'invoice.delete' && !d.is_trusted) {
    console.warn('[security] refused synthetic delete click');
    e.preventDefault();
  }
});

リファレンスデモ example-v20-full.php には、パネル出力でその違いを示すボタンペア（v20_panel.istrusted_real と v20_panel.istrusted_fake）が含まれています。

HMAC マニフェスト署名

サーバー側で署名を生成します。

import hmac, hashlib, json
manifest_body = json.dumps(manifest, sort_keys=True, separators=(',', ':'))
sig = hmac.new(
    tenant_secret.encode('utf-8'),
    manifest_body.encode('utf-8'),
    hashlib.sha256
).hexdigest()
manifest['provenance'] = {
    'signed_at': now_iso8601(),
    'signed_by': tenant_slug,
    'signature': sig
}

クライアント側:

NAC.set_provenance_secret(SECRET_FROM_AUTHED_RESPONSE);
NAC.register(manifest);
// runtime recomputes hmac, rejects if mismatch

シークレットは認証済みサーバーレスポンスから取得しなければなりません。JS ソースに埋め込まないでください。脅威モデルが要求する場合は、セッションごとにローテーションしてください。

脆弱性の報告

nac@yujin.dev に以下の内容をメールで送信してください。

  1. 脆弱性の説明。
  2. 再現手順または概念実証（PoC）。
  3. 影響を受ける NAC3 のバージョン。
  4. 緩和策の提案（ある場合）。

公開 GitHub Issue は作成しないでください。私たちは以下を約束します。

  • 3 営業日以内に受領を確認します。
  • 10 営業日以内にトリアージ評価を提供します。
  • 報告者と開示タイミングを調整します。

公開仕様に影響するクリティカルな問題は 30 日以内にパッチリリースで対応します。それ以下の重大度の問題は 90 日以内に対応します。

NAC3 が明示的に行わないこと

  • ユーザーの認証。
  • 転送中のデータの暗号化（TLS を使用してください）。
  • ユーザーが許可された操作を行うことの防止。
  • エージェント同士の分離（すべて同一ページで動作します。分離が必要な場合は別々のページで実行してください）。
  • 個別アクションへの署名（マニフェストのみ）。アクションごとの署名は v3.0 の候補として追跡中です。
