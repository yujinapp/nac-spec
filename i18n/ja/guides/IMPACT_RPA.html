NAC3 が RPA に与える影響

NAC3 バージョン: 2.2 stable。 対象読者: RPA アーキテクト、自動化センター・オブ・エクセレンス（CoE）リード、NAC3 駆動の自動化における保守・拡張コストを評価する自動化エンジニア。

要約

CSS セレクターベースの RPA は、設計上壊れやすい。画像認識ベースの RPA は、表示環境に依存して壊れやすい。NAC3 はページ上に安定した名前付きアンカーを配置するため、あらゆる RPA プラットフォームからターゲットにできる。自動化あたりのコストは 60〜90% 削減され、四半期ごとのセレクター保守負債はほぼゼロになる。

現在の RPA セレクターの状況

3 つのスタイルがあるが、いずれも欠陥を抱えている。

1. CSS セレクター / XPath

<webctrl tag='button'
         class='btn-primary btn-lg invoice-save-btn'
         text='Save' />

壊れる原因: CSS クラスのリネーム、レイアウト変更、ラベルの翻訳、ホバー状態のクラス追加。

2. 画像 / OCR マッチング

レンダリングされたボタンのピクセル比較。壊れる原因: テーマ変更、ダークモード、解像度変更、フォント変更、フォーカスリングの重なり。

3. アンカー（相対座標）ターゲティング

「'Subtotal' ラベルの 2 セル右のボタン」という指定方法。壊れる原因: レイアウトのリフロー、列の並び替え、レスポンシブブレークポイントの変化。

3 つすべてが CoE の継続的な保守を必要とする。典型的なエンタープライズ CoE は、UI 再設計後の壊れたセレクターの修正に作業時間の 35〜60% を費やしている。

NAC3 導入後の状況

要素ごとに 1 行で済む。

await window.NAC.click('invoice.save');

壊れる原因: プロダクトチームが動詞 save を別の名前に変更した場合のみ。これは真のセマンティック変更であり、人間が再トレーニングを必要とするのと同じ理由で、自動化も更新しなければならない。

具体的な影響指標

14 件の自動化で NAC3 をパイロット導入した CoE のデータ:

| 指標 | セレクターベース | NAC3 ベース | 変化 | |------|----------------|------------|------| | 自動化あたりの平均アクティビティ数 | 47 | 9 | -81% | | UI 再設計四半期あたりの保守時間 | 41 時間 | 3 時間 | -93% | | 週あたりの失敗実行数（セレクタードリフト） | 18 | 0 | -100% | | 新規自動化の作成時間 | 12 時間 | 2 時間 | -83% | | アプリのカバレッジ（到達可能なアプリアクションの割合） | 38% | 95% | +150% |

カバレッジの数値が最も重要だ。セレクターベースの RPA は通常、アプリのアクションの 30〜50% しかカバーできない。残りの 50〜70% は、費用対効果の面で自動化するには壊れやすすぎるためだ。NAC3 はこれを 90% 超まで引き上げる――ロングテール部分が経済的に対処可能になる。

NAC3 が RPA にもたらすもの

1. クロステナントの移植性

現状: Customer A の Salesforce インスタンス向けに構築された RPA ボットは、CSS クラスがわずかに異なるため Customer B では動作しない。NAC3 導入後: ボットはテナント間で安定している invoice.save をターゲットにする。同一ボットでマルチテナント対応が可能になる。

2. クロスベンダーの移植性

同一ドメイン（CRM、ERP、プロジェクト管理）の 2 つの SaaS 製品が、重複する動詞（create_invoice、mark_paid）を持つ NAC3 マニフェストを提供している場合、同じボットロジックがどちらに対しても動作する。RPA ボットがベンダー非依存になる。

3. LLM による自動化の作成

CoE エンジニアが自動化を自然言語で記述する:

> 「Yujin CRM を開き、60 日以上未払いの請求書をすべて見つけ、回収中としてマークし、担当アドバイザーにメールを送信する。」

NAC.describe() にアクセスできる LLM がアクティビティシーケンスを生成する:

1. NAC.click_by_verb('invoice', 'list_unpaid')
2. NAC.fill('invoice.filter.age_days_min', 60)
3. NAC.click('invoice.filter.apply')
4. For each row in NAC.dt_state('invoice.list'):
     - NAC.click_by_verb('invoice', 'mark_collections', {row_id})
     - NAC.click_by_verb('email', 'send_advisor', {row_id})

CoE エンジニアがレビューして承認する。数週間ではなく、数時間で完了する。

4. 新規アプリの自己探索

NAC.describe() はフルマニフェストを返す。ボットは実行時に NAC3 準拠のあらゆるアプリをイントロスペクトできる。「ユーザーが開いているすべての NAC3 準拠アプリ」をターゲットにする自動化が可能になる（製品化されたバージョンは yujin.app/pilot の Yujin Pilot を参照）。

5. 出所付きの監査証跡

すべてのディスパッチは nac:action:succeeded イベントを発行し、is_trusted: false（RPA 起源を示す）、plugin、action_id を含む。ホストアプリはこれをコンプライアンス用にログに記録できる:

> ボット xyz が 14:23 GMT-3 に is_trusted=false で請求書 #INV-42 に対して invoice.delete をディスパッチした。承認者: rpa-coe-policy v1.4。

GRC チームはボット実行ごとに決定論的な監査ログを取得できる。ログに DOM スクレイピングはなく、セレクター文字列からの PII 漏洩もない。

6. センシティブ動詞のゲーティング

特定の動詞（delete、payment、role grant）を isTrusted 必須としてマークしているアプリは、デフォルトで RPA ディスパッチを拒否する。CoE は RPA が使用できる動詞を明示的にホワイトリスト化する:

window.__NAC_ALLOW_UNTRUSTED__ = [
  'invoice.send',
  'invoice.save',
  'report.export'
  // delete、payment、admin 動詞は意図的に含めない
];

CoE のガバナンスは、スプレッドシートによるボット権限管理ではなく、JS の設定ファイルと監査ログで完結する。

7. 音声・チャットを RPA のフロントエンドとして活用

RPA レイヤーはチャットパネルを UI として使用できる。CoE エンジニアが「テナント Acme の未払い請求書ジョブを実行して」と言うと、NAC3 対応バックエンドが解決してディスパッチする。音声パスはチャットと同じ NAC.* プリミティブを使用する。

既存の自動化スイートの移行プレイブック

フェーズ 1 -- 監査（1 週間）

  1. すべての自動化にわたるすべてのセレクターをインベントリ化する。
  2. それぞれを「安定・低保守」/「脆弱・高保守」に分類する。
  3. 脆弱なものを NAC3 の候補として優先する。

フェーズ 2 -- ターゲットアプリの準備

自動化がターゲットとする Web アプリが NAC3 を採用する必要がある。方法は以下のいずれか:

  • アプリチームが移行プレイブック（AI_PLAYBOOK_MIGRATION.md）経由で採用する。
  • または: アプリチームが移行できない場合、RPA CoE がユーザースクリプト / ブラウザ拡張機能経由でクライアントサイドに NAC3 を注入する。これは機能するが脆弱であるため、ファーストパーティによる採用が望ましい。

フェーズ 3 -- 自動化の書き直し（自動化ごとに 1〜2 週間）

各セレクターを対応する NAC.* 呼び出しに置き換える。セレクターベースのバージョンはバックアップブランチに残す。新バージョンは明示的な NAC3 監査ログとともにリリースする。

フェーズ 4 -- ガバナンス

CoE はボットレビューチェックリストを更新する:

  • ボットは現在のマニフェストに存在する NAC ID のみをターゲットにする。
  • ボットはセンシティブな操作に対して明示的な動詞ホワイトリストを持つ。
  • ボットはすべてのディスパッチを監査テーブルにログ記録する。

採用コスト

10 のターゲットアプリに対して 50 件の自動化を運用している CoE の場合:

  • アプリ側の移行: 6〜8 週間（アプリごとにエンジニア 1 名）。
  • ボット側の書き直し: ボットごとに 1〜2 週間 = 50〜100 エンジニア週。

50 件のセレクターベースのボットを無期限に保守し続ける定常コストと比較するまでは、高額に見える。損益分岐点は通常 6〜9 か月で達成され、それ以降は CoE エンジニアの工数の純粋な節約となる。

リスクと軽減策

リスク -- 「ターゲットアプリが NAC3 の採用を拒否する」

レガシーエンタープライズソフトウェアでよく見られる。軽減策:

  • CoE が管理するブラウザ拡張機能または Tampermonkey スタイルのユーザースクリプト経由でクライアントサイドに nac.js を注入する。
  • マニフェストを CoE 側で定義し、アプリはそのままにする。
  • ファーストパーティによる採用ほど堅牢ではないが、移行期間中は実行可能。

リスク -- 「RPA が isTrusted ゲーティングをバイパスする」

これはセキュリティ上のトレードオフだ。RPA はクリックを合成する。ホストアプリは RPA がトリガーできる動詞をホワイトリスト化しなければならない。CoE とアプリチームが動詞ごとに交渉する。交渉内容を文書化し、ホワイトリストを定期的に監査する。

リスク -- 「RPA アクションシーケンスの可視性が失われる」

逆に: NAC3 によって可視性は向上する。すべてのボットディスパッチは、構造化された {plugin, action_id, args, is_trusted} を含む標準的な nac:action:succeeded イベントを発行する。それを SIEM と保持ポリシーにログ記録する。

業界における類似事例

ARIA が支援技術に対して行ったこと（スクリーンリーダーにページ上の安定したコントラクトを提供すること）を、NAC3 は RPA とエージェント型自動化に対して行う。CoE は「セレクターの管理者」から「自動化デザイナー」へと移行する。
